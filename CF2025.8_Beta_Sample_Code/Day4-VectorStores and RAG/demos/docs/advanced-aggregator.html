Content aggregator

Merge result lists from multiple retrievers into one ordered context block.

The problem it solves

Put every document into one store and a single retriever will return the top‑K vector matches. For a question that spans several domains, the top‑K chunks tend to cluster on whichever domain has the strongest embedding match — the other domains get starved out of the context.

Once you split your corpus into domain‑specific stores (HR, Product, FAQ, …), each store gets its own retriever. The query router sends the query to all of them (or a routing model picks a subset), and every queried retriever returns its own ranked list. The contentAggregator is what takes those parallel lists and collapses them into the single ordered list that ends up in the prompt.

What this panel does

  • Left (before) — one vector store for every document, one retriever, topK=5. No aggregator involved.
  • Right (after) — two stores (HR & Product/FAQ), each with its own retriever pulling topK=3. contentAggregator merges both lists with a visible source boundary between chunks and a token cap on the combined context.

The “Retrieved context” panel on the right shows two labeled sections (one per store) so you can visibly see the aggregator pulling from both sources.

The code

rag = agent({
    chatModel : chatModel,
    retrievalAugmentor : {
        queryRouter : {
            // No routingModel => query goes to BOTH retrievers
            contentRetrievers : [
                { vectorStore: hrStore,      maxResults: 3, description: "HR store" },
                { vectorStore: productStore, maxResults: 3, description: "Product + FAQ" }
            ]
        },
        contentAggregator : {
            separator : "\n--- source boundary ---\n",
            maxTokens : 1200
        }
    }
});

Aggregator fields

FieldUsed for
separatorString inserted between chunks in the merged context — useful for visibly grouping chunks from different retrievers.
maxTokensToken ceiling for the combined context block handed to the injector.
typeSelects the aggregator strategy (default / reranking / custom).
implementationA custom component instance if you want to bypass the built‑in strategies.

When to add one

  • You have more than one retriever and you want the combined context capped with maxTokens.
  • You want chunks from different retrievers visibly separated so the LLM can tell sources apart.
  • You want to plug in a custom aggregator class via implementation for deterministic ordering, de‑duplication, or cross‑source re‑ranking.
